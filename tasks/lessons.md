# Lessons

## Routing

- SloaneVault remains the canonical infrastructure knowledge base. Use `/home/cvsloane/SloaneVault/tasks/lessons.md`, `tasks/decisions.jsonl`, and the VPS infrastructure runbooks for service facts, cutover decisions, DNS, secrets policy, and incident history.
- Keep infra-dashboard lessons focused on product/code behavior: collector assumptions, health-check semantics, alert thresholds, service inventory modeling, display copy that can mislead operators, and deployment quirks specific to this dashboard.
- When a SloaneVault infrastructure lesson changes what the dashboard should collect or display, mirror only that dashboard-relevant rule here and link back to the vault source instead of copying the full runbook.
- For home-network client diagnostics, do not imply that two clients must be on the same radio for local discovery tools to work. What matters for Flow-style peer discovery is same reachable subnet/broadcast domain with client isolation off; 2.4/5/6 GHz radios on the same bridge should be treated as one LAN for dashboard checks.
- Do not treat a failed OpenWrt/GL.iNet 6 GHz SSID move as an equipment limitation until the generated hostapd bridge, live `bridge link` state, and DHCP lease path are checked. If 6 GHz clients join the right SSID but receive the wrong subnet, investigate stale/generated bridge binding or reload order first.
- Home-network endpoint data should ride on the existing `/home-network` snapshot as optional `windows_laptops` records. The dashboard must tolerate the merge file being absent so router monitoring does not depend on Windows laptop SSH.
- After changing home-network L2/DHCP topology, do not call the migration complete from synthetic router DHCP tests alone. Verify named client devices through association, Main DHCP lease, ARP/neigh reachability, and recent logs; update the collector if remote AP station rows need to be joined against centralized Main leases.
- Before saying a home MoCA/backhaul path cannot carry VLANs, prove the OpenWrt switch/CPU-port model is correct. On GL.iNet Flint-style `swconfig` routers, inbound tagged traffic may require explicit switch VLAN membership plus testing on the CPU-facing VLAN interface; use packet capture and bidirectional `arping`, not only ICMP pings from an interface outside the normal firewall zones.
- When diagnosing a named home laptop after Wi-Fi changes, do not infer Windows service state from closed ports until the device has a fresh lease and answers ARP. Treat "associated to Wi-Fi but no ARP/DHCP renewal" as a Wi-Fi/IP attachment problem first, and keep each named laptop's status separate.
- On QCA/GL.iNet Flint routers, do not assume UCI wireless changes are live until the generated `/tmp/run/hostapd-wlan*.conf` files are checked. `wireless.qcawifi.wds_ext` can emit `wds_sta=1` independently of per-SSID `wds=0`, and a real Wi-Fi restart may be required.
- For same-SSID multi-AP layouts, validate the RF plan after the L2 design is fixed. Three APs on the same 5 GHz 80 MHz block can create roaming/stale-station failures even when VLANs and DHCP are correct; prefer non-overlapping channels and verify with named-client ARP/service checks.
- On Flint/OpenWrt, `hostapd_cli` may need the per-radio control path such as `/var/run/hostapd-wifi1`; a bare `hostapd_cli -i wlan11` can report nothing useful and leave stale stations in place.
- Do not conclude a named Windows laptop is asleep after the operator confirms it is awake. If ARP reaches the laptop but TCP management ports time out, prove packet direction with router-side capture; delivered SYNs with no reply indicate Windows firewall/service/power profile state, while no ARP reply from an associated client points back to AP/client attachment.
- For older Windows laptops that flap on `Home-K` 5 GHz, prefer a targeted per-client band pin over disabling an entire SSID. Use `hostapd_cli`/UCI MAC deny on the problematic BSSID, then verify the named client by AP association, fresh DHCP lease, ARP, and service-port tests.
- When the operator says they are handling a Windows endpoint install, stop chasing that endpoint and return to the requested network-architecture question. Keep endpoint install status separate from router/AP topology status.
- For home-network dashboard modeling, distinguish mobility classes. Tablets/phones need whole-house AP roaming quality; fixed laptops and gaming PCs can be pinned to a stable AP/band/network. Do not treat every `Home-K` endpoint as needing seamless multi-AP roaming.
- For home-network roaming health, flag same-MAC multi-AP stale station rows and verify cross-router reachability. A client associated to a remote AP is not healthy unless Main, Office, and School can all reach it; bridge netfilter on remote APs can break L2 transit even when the local AP can ping the client.
- For bridged home-network SSIDs with centralized DHCP, enrich all AP station rows from the combined lease set before rendering device identity or subnet health. Remote AP clients can have their live DHCP lease only on Main, so per-router lease merging can make known devices appear unknown or stale.
- For home-network dashboard wording, distinguish policy/subnet authority from physical AP placement. Do not imply that Main DHCP or Main-only policy means every device should attach to the Main/Cabinet radio; Office and School are expected coverage APs when the selected SSID/path is stable.
- For kids school laptops, do not suggest `Heaviside` as the normal placement just because endpoint NextDNS exists. These devices require `Home-K` or an equivalent isolated kids subnet for quarantine; endpoint controls are defense-in-depth, not a substitute for network isolation.
- For the fixed kids laptop rollout, treat `Home-K`, `Home-K-School`, and `Home-K-Office` as kids-policy SSIDs in dashboard summaries, but keep expected SSIDs role-specific per router so Office/School do not falsely warn for missing Main-only SSIDs.
- When migrating a Windows laptop between fixed kids SSIDs, keep an existing management path active until the new path proves both Wi-Fi quality and TCP management reachability. A successful `netsh wlan connect` plus AP association is not enough if SSH later times out; verify ARP, service ports, and Windows firewall/network-profile behavior before calling the adapter choice final.
- Do not label a Windows Wi-Fi adapter as globally bad from one SSID/AP result. Chloe's TP-Link USB adapter was terrible on Main `Home-K` 2.4 GHz but became the best path on fixed Office `Home-K-Office`; compare adapter plus SSID plus BSSID before choosing the client baseline.
- When the operator explicitly excludes a device from a home-network remediation pass, remove that device from the active fix set even if telemetry still shows it as noisy or policy-drifted. Document the exclusion and continue with the remaining routers, laptops, dashboards, and monitoring fixes.
- For dashboard-backed home-network logs, do not require router-pushed syslog freshness before preserving useful router event evidence. Persist SSH-pulled `logread` tails with per-router heartbeats, and present pushed syslog as a secondary transport unless the monitoring host has a stable reachable management address.
- For Tuya-OUI MACs that churn on 2.4 GHz, treat smart plugs/power strips as a likely identity before escalating to laptop/tablet policy work. Contain RF noise if needed, but do not classify the device as a managed school or business endpoint without stronger evidence.
- Before changing home-network policy, verify the live School/Office router layout and client subnets from current telemetry. Do not assume old `Home-K` architecture language still describes the active network; map actual clients to `Heaviside`, `Home-K-School`, `Home-K-Office`, and the routed School/Office subnets first.

## 2026-04-24
- Do not treat aggregate Hermes fleet health or `/api/health` as proof that all dashboard-backed remediation paths are healthy. When the dashboard shows something down, inspect the specific source behind the issue row: AutoHEAL heartbeat/status, worker supervisor, Alertmanager, site health, Postgres/PgBouncer container probes, and the relevant host service logs.
- AutoHEAL health depends on three separate runtime facts: the systemd timer/service, the Coolify API URL/token, and Redis heartbeat writes. A worker can exit successfully after fixing the Coolify URL while still failing to update dashboard-visible state if Redis credentials or the deployed worker script are stale.
- Site health intentionally reads Coolify DB rows with non-empty FQDNs, so retired or internal-only Coolify applications can show as "down" even when no public app should be restored. Before restarting anything, check the app row `status`, DNS resolution, and whether a container exists; stale apps should be excluded or retired rather than auto-redeployed.
- Worker health has two separate dashboard counters: BullMQ queues without `stalled-check` heartbeats and the host worker-supervisor status key. Debug both sources separately before saying workers are down, and keep the supervisor script tolerant of PM2 non-JSON startup output so one bad collector does not erase the whole status report.

## 2026-04-09
- For cross-host Coolify builds, a successful remote build on the build server is not enough; if rollout happens on a different host, plan the image transport path explicitly and validate registry push/pull credentials before treating the deploy path as complete.
- When onboarding a remote Coolify build server that connects as `root`, ensure `/root/.docker/config.json` exists on that host before testing build-server bootstrap behavior.
- When a build/deploy issue first shows up on one app, confirm whether the intended fix is app-specific or a new platform default; for Coolify build-server work here, treat `homelinux` as shared infrastructure for all apps unless the user explicitly scopes it narrower.
- In the current Coolify version here, multiple reachable build servers are load-balanced with `random()`, not primary/backup ordered. To make one server primary and the deployment host the backup, keep only the intended primary in the build-server pool and rely on the deployment server fallback when no build server is available.
- Coolify `dockercompose` applications do not go through the same build-server image push/pull path as `dockerfile` and `nixpacks` applications in this version. Validate compose behavior separately before claiming fleet-wide build-server coverage.
- Dashboard-facing Coolify capacity should distinguish `server_settings.concurrent_builds` from Horizon worker process capacity. `homelinux` build concurrency comes from Coolify DB, while the deployment worker pool comes from the Coolify container env (`HORIZON_MAX_PROCESSES`) and must be mirrored into infra-dashboard env (`COOLIFY_DEPLOYMENT_WORKERS_*`) if the UI displays it.
- Do not assume a staging app is an acceptable validation target for platform changes. Confirm whether the intended production policy is `production-only` first; if staging should be retired, validate against production configuration without keeping a live staging app around.
